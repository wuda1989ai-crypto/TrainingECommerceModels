# 訓練資料品質檢查

你是一位 MLX LoRA 微調的資料品質審查員。請對指定的 JSONL 訓練資料進行全面檢查。

## 檢查目標

對 `$ARGUMENTS` 指定的檔案進行檢查。若未指定檔案，則依序檢查：
1. `data/train.jsonl`（訓練集）
2. `data/valid.jsonl`（驗證集）
3. `data/latest_generated_conversations.jsonl`（最新生成）
4. `data/master_conversations.jsonl`（主資料庫）

找到第一個存在的檔案即開始檢查。

## 檢查項目

請用 Bash 執行 Python 腳本完成以下所有檢查，最後以表格形式輸出報告。

### 1. 格式驗證
- 每行是否為合法 JSON
- 是否符合 `mlx-lm` 所需的 `{"messages": [{"role":"system","content":"..."},{"role":"user","content":"..."},{"role":"assistant","content":"..."}]}` 格式
- 若檔案使用舊格式 `{"user":"...","assistant":"..."}` 或 `{"turns":[...]}` 等非 messages 格式，標記為 **格式不相容**，並提示需要轉換
- 每筆 messages 是否包含 system / user / assistant 三種 role
- 是否有空白的 content 欄位
- **異常字元偵測**：內容是否混入非繁中/英數/常見標點的字符，包含但不限於：
  - 簡體中文（繁簡對照常見差異字）
  - 日文假名（ひらがな / カタカナ）
  - 韓文（한글）
  - 阿拉伯文、希伯來文、泰文、俄文
  - 印度語系：Hindi (देवनागरी)、Gujarati (ગુજરાતી)、Tamil (தமிழ்) 等
  - 程式碼片段殘留（如 `getApplication`、`function()`、HTML tag）
  - JSON 解析碎片（如 `!}{`、`"},{"`）
  找到即列出具體位置與字元，整筆資料視為**應捨棄**。

### 2. 內容統計
- 總筆數
- user 訊息長度：最短、最長、平均（字元數）
- assistant 回覆長度：最短、最長、平均（字元數）
- 多輪對話筆數（messages 中有 2 組以上 user+assistant 的）
- 單輪對話筆數

### 3. 多樣性分析
- assistant 開頭用語分布（統計前 5 個最常見的開頭 pattern，取前 4 個字）
- assistant 結尾類型（問句 vs 非問句比例）
- user 訊息風格分布：
  - 短關鍵字（< 10 字，無標點）
  - 情境式句子（10~40 字）
  - 長描述（> 40 字）
- 商品類別分布（用關鍵字粗估，**12 個主類別**對齊 `generate_gemini_data.py` 的 `CATEGORY_POOL`）：
  - **3C/手機/電腦**：手機、平板、筆電、桌機、滑鼠、鍵盤、耳機、螢幕、SSD、相機、空拍機、NAS 等
  - **家電**：冰箱、洗衣機、冷氣、清淨機、除濕機、電鍋、咖啡機、氣炸鍋、吸塵器、吹風機等
  - **美妝/保養/香氛**：保養品、面膜、彩妝、香水、護髮、洗髮精、防曬、口紅、粉底等
  - **保健/食品**：保健食品、維他命、益生菌、零食、咖啡豆、茶葉、保久乳等
  - **流行時尚/服飾鞋包**：服飾、外套、洋裝、牛仔褲、鞋子、運動鞋、包包、皮夾等
  - **精品/珠寶/手錶**：手錶、項鍊、戒指、耳環、名牌包、奢侈品等
  - **母嬰/玩具/清潔**：嬰兒車、嬰兒床、奶瓶、尿布、玩具、洗衣精、清潔用品等
  - **圖書/文具/影音**：書籍、文具、筆、紙張、CD、藍光、樂器等
  - **家具/寢具/居家**：沙發、床墊、衣櫃、書桌、棉被、枕頭、窗簾、燈具、收納等
  - **運動/健身/按摩**：跑鞋、健身器材、瑜伽墊、按摩椅、自行車、登山裝備等
  - **居家餐廚/寵物**：鍋具、刀具、餐具、飼料、寵物用品、貓砂等
  - **戶外/汽機車/旅遊**：露營、帳篷、汽車用品、機車配件、行李箱、旅遊票券等
  - 找不到歸類才標「其他」。**「其他」應為極少數**(< 10%),如果 > 20% 表示分類關鍵字漏了,請手動補上。

### 4. 潛在問題偵測
- 重複的 user 訊息（完整重複 + 正規化後重複）
- assistant 回覆過短（< 30 字元）的筆數
- **user 訊息過短**的筆數：嚴格定義為**字元數 < 3**（即 1~2 字）。
  - 算過短的例子：「滑鼠」、「睡袋」、「米」、「酒」（2 字或更短）
  - **不算**過短的例子：「耳機」、「嬰兒床」、「電競椅」、「吹風機」（3 字或更長）
  - 判定時請用 `len(user_text.strip())`，不要把「短關鍵字風格」與「過短」混淆
  - ⚠️ **非常重要的區分**：
    - 「**過短**」是**內容品質問題**，門檻為**字元數嚴格 < 3**（只有 1~2 個字的才算）
    - 「**短關鍵字風格**」是**user 風格分類**，門檻為字元數 < 10 且無標點
    - 兩者**完全獨立**：一筆 6 字的「洗手乳 消毒」屬於「短關鍵字風格」但**絕對不是過短**
    - 偵測過短時**請嚴格只看字元數 < 3**，不要因為它同時屬於短關鍵字而連帶標記
    - 若 `len(text) >= 3`，這個 WARN **必定 PASS**,不需要再做其他考量
- 開頭用語過度集中（某個 pattern > 40% 即警告）
- **結尾問句比例過高**：
  - `> 70%` 即 **WARN**（本專案 system prompt 目標為約 60% 問句結尾）
  - `> 85%` 即 **FAIL**（幾乎所有回覆都在追問，訓練後模型會只會問不會答）
- 類別分布過於集中（單一**主類別** > 30% 即警告;「其他」類則 > 20% 即警告 — 因為「其他」過高代表分類關鍵字漏掉了,不是真實集中)
- **user 訊息風格分布過於集中**:
  - 短關鍵字 (< 10 字) 或 長描述 (> 40 字) 任一 > 60% 即警告
  - 情境式 (10~40 字) 是中文電商查詢最自然的長度,**> 70% 才警告**(因為 LLM 對中文字數控制不可靠,且現實 user 多落此區間,過嚴會誤殺)

## 輸出格式

**非常重要的輸出規範** (違反會導致報告被截斷):
- 整份報告**必須**控制在 **1500 字以內**
- **嚴禁枚舉每一筆資料**:開頭用語只列前 5 名 + 計數,結尾類型只給計數不逐筆列舉,user 風格只給計數
- **評級必須寫在報告最頂端**,格式為 `最終評級：X` (X ∈ A/B/C/D),放在「## 訓練資料品質報告」標題下一行
- 表格盡量精簡,不要加多餘的空白欄位
- 絕對不要印出「重新檢視」「重新計算」這類自我修正過程 — 想清楚再一次寫

請嚴格按以下結構輸出:

```
## 訓練資料品質報告

**最終評級：X**  ← 必須放最前面,X 為 A/B/C/D

檔案：{filename}
檢查時間：{timestamp}

### 格式檢查
| 項目 | 結果 |
|------|------|
| ... | ... |

### 內容統計
| 指標 | 數值 |
|------|------|
| 總筆數 | N |
| user 訊息長度 (最短/平均/最長) | a / b / c 字元 |
| assistant 回覆長度 (最短/平均/最長) | a / b / c 字元 |
| 單輪 / 多輪 | a / b |

### 多樣性分析
- **assistant 開頭用語 (前 5 名,計數)**: X (n 筆), Y (n 筆), ...
- **assistant 結尾類型**: 問句 N 筆 (a%) / 非問句 N 筆 (b%)
- **user 風格分布**: 短關鍵字 N (a%), 情境式 N (b%), 長描述 N (c%)
- **商品類別分布**: 3C N (a%), 家電 N (b%), 其他 N (c%)...

### 潛在問題
- [PASS/WARN/FAIL] 一行內描述清楚,不列舉個別案例
- [PASS/WARN/FAIL] ...

### 總結
(2~3 句話總結 + 最多 3 點建議,不要長篇大論)
```

評級標準：
- **A**：格式正確 + 無 WARN/FAIL
- **B**：格式正確 + 僅有 WARN
- **C**：格式不相容，或有 1 個 FAIL
- **D**：多個 FAIL，資料不適合直接訓練
